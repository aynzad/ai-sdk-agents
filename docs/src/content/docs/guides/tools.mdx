title: Tools
description: Give agents capabilities with function tools, client-side tools, agent-as-tool, and guarded tools.

Tools extend what an agent can do. The library supports four kinds of tools: **function tools**, **client-side tools** (human-in-the-loop), **agent-as-tool**, and **guarded tools**.

## Function Tools

Function tools are standard Vercel AI SDK tools defined with a Zod schema and an execute function:

```ts
import { Agent, Runner } from "@aynzad/ai-sdk-agents";
import { openai } from "@ai-sdk/openai";
import { z } from "zod";

const agent = new Agent({
  name: "Assistant",
  model: openai("gpt-4o"),
  instructions: "You help with math and weather.",
  tools: {
    calculate: {
      description: "Evaluate a math expression",
      parameters: z.object({ expression: z.string() }),
      execute: async ({ expression }) => String(eval(expression)),
    },
    getWeather: {
      description: "Get the weather for a city",
      parameters: z.object({ city: z.string() }),
      execute: async ({ city }) => `72°F and sunny in ${city}`,
    },
  },
});
```

Tools are passed directly to AI SDK's `generateText` / `streamText`. The model chooses which tool to call based on the user's message.

## Client-Side Tools (Human-in-the-Loop)

Client-side tools have no `execute` function. When the model invokes one, the stream pauses and waits for the frontend to provide a result — enabling human approval flows, confirmations, or any UI-driven tool resolution.

Define client-side tools in `clientTools` and use `Runner.streamUI()` to serve them:

```ts
import { tool } from "ai";
import { Agent } from "@aynzad/ai-sdk-agents";
import { openai } from "@ai-sdk/openai";
import { z } from "zod";

const getRecord = tool({
  description: "Look up a record by ID",
  inputSchema: z.object({ id: z.string() }),
  execute: async ({ id }) => ({ id, name: "Acme Corp", status: "active" }),
});

const updateRecord = tool({
  description: "Update a record. Requires human approval.",
  inputSchema: z.object({
    id: z.string(),
    field: z.string(),
    value: z.string(),
  }),
  // No execute — resolved by the frontend
});

const agent = new Agent({
  name: "Database Agent",
  model: openai("gpt-4o"),
  instructions: "Look up and update records. Updates require approval.",
  tools: { getRecord },
  clientTools: { updateRecord },
});
```

In your Next.js API route, use `Runner.streamUI()` instead of `Runner.stream()`:

```ts
import { type UIMessage } from "ai";
import { Runner } from "@aynzad/ai-sdk-agents";

export async function POST(req: Request) {
  const { messages } = (await req.json()) as { messages: UIMessage[] };
  return Runner.streamUI(agent, messages);
}
```

On the frontend, use `useChat` with `addToolOutput` to resolve client-side tool calls:

```tsx
import { useChat } from "@ai-sdk/react";
import { lastAssistantMessageIsCompleteWithToolCalls } from "ai";

const { messages, sendMessage, addToolOutput } = useChat({
  sendAutomaticallyWhen: lastAssistantMessageIsCompleteWithToolCalls,
});

// When the model invokes updateRecord, render an approval UI and call:
addToolOutput({
  tool: "updateRecord",
  toolCallId: callId,
  output: JSON.stringify({ success: true, message: "Approved" }),
});
```

### tools vs clientTools

| Property | Execution | Use case |
|----------|-----------|----------|
| `tools` | Server-side (auto-executed) | Database queries, API calls, computations |
| `clientTools` | Client-side (via `addToolOutput`) | Human approval, confirmations, UI input |

Both are available to the model simultaneously — the model chooses which to call based on the conversation. `Runner.streamUI()` merges them into a single `streamText()` call.

### What Runner.streamUI() supports

`Runner.streamUI()` applies the agent's full configuration before calling `streamText()`:

- Dynamic instructions resolution
- Input guardrails
- Model settings (temperature, maxOutputTokens, etc.)
- Agent hooks (`onStart`, `onEnd`)
- Tool guardrails on server-side tools

It does **not** support handoffs or multi-turn agent loops — use `Runner.stream()` for those. See [Streaming](/ai-sdk-agents/guides/streaming/) for details.

## Agent as Tool

Wrap an agent as a tool that another agent can invoke. The parent agent **retains control** after the sub-agent completes:

```ts
const factChecker = new Agent({
  name: "FactChecker",
  model: openai("gpt-4o"),
  instructions: "Verify claims and return true/false with reasoning.",
});

const writer = new Agent({
  name: "Writer",
  model: openai("gpt-4o"),
  instructions: "Write articles. Use the fact checker for any claims.",
  tools: {
    checkFact: factChecker.asTool({
      description: "Verify a factual claim",
    }),
  },
});
```

When the LLM calls `checkFact`, the library internally runs `Runner.run(factChecker, message)` and returns the output as the tool result. The parent agent then continues with this information.

### Agent-as-Tool vs Handoffs

| Feature | Agent as Tool | Handoff |
|---------|--------------|---------|
| Control | Parent retains control | Target takes over completely |
| History | Sub-agent gets only the tool message | Target gets full conversation history |
| Use case | Delegation for a specific task | Routing to a specialist |

## Guarded Tools

Guarded tools wrap standard tools with input and output guardrails that run on every invocation:

```ts
import { guardedTool, defineToolInputGuardrail, ToolGuardrailBehaviorFactory } from "@aynzad/ai-sdk-agents";
import { z } from "zod";

const noSQLInjection = defineToolInputGuardrail({
  name: "no-sql-injection",
  execute: async ({ args }) => {
    if (typeof args.query === "string" && args.query.includes("DROP TABLE")) {
      return ToolGuardrailBehaviorFactory.rejectContent(
        "Query rejected: potential SQL injection detected."
      );
    }
    return ToolGuardrailBehaviorFactory.allow();
  },
});

const dbQuery = guardedTool({
  description: "Run a database query",
  parameters: z.object({ query: z.string() }),
  execute: async ({ query }) => `Results for: ${query}`,
  inputGuardrails: [noSQLInjection],
});
```

### Guardrail Behaviors

Tool guardrails return one of three behaviors:

- **`allow()`** — proceed with execution
- **`rejectContent(message)`** — skip execution, return the message as the tool result (model sees it and can react)
- **`throwException(reason?, metadata?)`** — throw `ToolGuardrailTripwiredError`, halting the run

```ts
import { ToolGuardrailBehaviorFactory } from "@aynzad/ai-sdk-agents";

ToolGuardrailBehaviorFactory.allow();
ToolGuardrailBehaviorFactory.rejectContent("Not allowed.");
ToolGuardrailBehaviorFactory.throwException("Critical violation");
```

### Output Guardrails on Tools

Validate what a tool returns:

```ts
import { defineToolOutputGuardrail, guardedTool, ToolGuardrailBehaviorFactory } from "@aynzad/ai-sdk-agents";
import { z } from "zod";

const noPII = defineToolOutputGuardrail({
  name: "no-pii-in-output",
  execute: async ({ output }) => {
    if (typeof output === "string" && output.includes("SSN")) {
      return ToolGuardrailBehaviorFactory.rejectContent(
        "Output redacted: contains PII."
      );
    }
    return ToolGuardrailBehaviorFactory.allow();
  },
});

const lookupUser = guardedTool({
  description: "Look up user information",
  parameters: z.object({ userId: z.string() }),
  execute: async ({ userId }) => `User ${userId}: John Doe, SSN: 123-45-6789`,
  outputGuardrails: [noPII],
});
```

### Difference from Agent Guardrails

| Feature | Agent Guardrails | Tool Guardrails |
|---------|-----------------|-----------------|
| Scope | Entire agent input/output | Per-tool invocation |
| Execution | Parallel | Sequential |
| Failure mode | Always throws `GuardrailTripwiredError` | `allow`, `rejectContent`, or `throwException` |
| Attached to | Agent config | Individual tool via `guardedTool()` |

See [Guardrails](/ai-sdk-agents/guides/guardrails/) for agent-level guardrails.
